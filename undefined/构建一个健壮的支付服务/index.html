构建一个健壮的支付服务

Posted by humSuperman
发布于 2022-07-13
阅读时长 3 min
本文字数： 931

业务背景

因公司业务模式、财务合规及税务审查需求，需要引入分帐模式。
存在多个业务端，未来也可能开发新的c端应用。
故将支付服务单独开发，只涉及订单的支付、退款、分帐功能，无需关注业务模式。
因商户入网、交易分帐、提现、交易渠道不具有代表性，再次不做赘述。
本文只关注交易、退款两大模块。

接口梳理

getAccountOrCreate

  • 获取账户，不存在则创建
  • 请求方式：get
  • 入参：
    • mobile : 用户手机号
    • nickName : 用户昵称
  • 业务逻辑：
    • 使用手机号作为逻辑主键，查询用户是否存在，不存在创建用户、店铺
  • 返回：
    • userId
    • shopId

getSellerId

  • 获取收单店铺id
  • 请求方式：get
  • 入参：
    • source : 业务类型，在支付平台有有一个静态类或数据库表在维护
  • 业务逻辑：
    • 根据入参source，查询对应的收款主体shopId
    • 设置一个备份收款shopId，无论如何不要让前端用户无法支付
    • 在source没有配置对应收款shopId时，返回默认配置
  • 返回：
    • shopId

tradePayment

  • 创建支付单并拉起支付
  • 请求方式：post
  • 入参：
    • outerId : 业务层订单编号
    • source : 业务类型
    • buyerId : 用户shopId
    • sellerId : 收单shopId
    • amount : 订单支付金额
  • 业务逻辑：
    • 创建一个交易trade
    • 在trade_source绑定tradeId与outerId
    • 将tradeId加入redis，定时在支付第三方支付平台查询支付结果
    • 拉起第三方收银台
  • 返回：
    • 收银台信息

getTradePaymentResult

  • 获取订单支付结果
  • 请求方式：get
  • 入参：
    • tradeId : 交易单号
  • 业务逻辑：
    • 查询trade详情
    • 查询trade绑定trade_source
  • 返回：
    • trade
    • trade_source

以上接口等返回信息均为成功状态
若接口报错需返回错误信息

tradeRefund

  • 订单退款
  • 请求方式：post
  • 入参：
    • tradeId : 交易单号
    • refundAmount : 退款金额
  • 业务逻辑：
    • 查询trade详情，校验支付状态
    • 查询订单已退款总金额，包含退款中、退款成功
    • 保存refund并请求支付平台退款
    • refundId加入redis，定时在支付第三方支付平台查询退款状态
    • 更新退款状态
  • 返回：
    • refundId
    • refundAmount

定时任务

tradResult

  • 订单支付结果
  • 数据结构：
    • redis zset
  • 命令：
    • zadd trade:result:zset Unix-timestamp tradeId
  • 任务流程
    • 获取24h内的tradeId，组装支付平台请求数据，逐条查询trade支付结果
    • 任务启动需要加锁，并增加过期时间，防止死锁；任务运行结束后移除锁
    • 更新trade支付状态时需要加锁，并增加过期时间，防止死锁；更新结束后移除锁
    • 更新trade之后在zset中移除tradeId
    • 移除24小时之前的所有tradeId

refundResult

  • 订单支退款结果
  • 数据结构：
    • redis zset
  • 命令：
    • zadd refund:result:zset Unix-timestamp refundId
  • 任务流程
    • 获取24h内的refundId，组装支付平台请求数据，逐条查询refund退款结果
    • 任务启动需要加锁，并增加过期时间，防止死锁；任务运行结束后移除锁
    • 更新refund退款状态时需要加锁，并增加过期时间，防止死锁；更新结束后移除锁
    • 更新refund之后在zset中移除refundId
    • 移除24小时之前的所有refundId

消息推送

将支付结果推送至具体的业务层，建议使用kafka、redis等消息队列
